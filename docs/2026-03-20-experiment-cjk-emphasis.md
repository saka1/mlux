# 実験記録: CJK テキストの emphasis 表現

日付: 2026-03-20

## 背景

mlux では Markdown の `*emphasis*` を Typst の `#emph[...]` に変換し、テーマ側の
`#show emph:` ルールで色変更のみ適用している。斜体フォント（italic）は Noto Sans JP に
存在しないため、和文テキストの emphasis は色だけで表現されている。

課題: 色変更だけで「強調」の意味が読み手に伝わるか不確実。

## 実験: 圏点（emphasis dots）アプローチ

JLReq（日本語組版の要件）に準拠し、CJK 文字に圏点（﹅ or •）を付ける方式を検討。
Typst の regex show rule で CJK 文字を検出し、`place` でドットをオーバーレイする。

### 実装

```typst
#show emph: it => {
  show regex("[\\p{Han}\\p{Hiragana}\\p{Katakana}ー]"): c => {
    box[#c#place(top + center, dy: -0.65em, text(size: 0.5em, "•"))]
  }
  text(style: "italic", fill: rgb("#b4befe"), it)
}
```

- CJK 文字: 文字上部に丸点を配置 + 色変更
- Latin 文字: イタリック体 + 色変更
- 混在テキスト: 文字単位で自動分離

### パラメータ探索

| バージョン | ドット文字 | サイズ | 位置 (dy) | 評価 |
|-----------|----------|--------|----------|------|
| v1 | ﹅ (ゴマ点) | 0.5em | -0.8em | 存在感が高すぎる。行間がくっついて見える |
| v2 | • (丸点) | 0.3em | -0.55em | 小さすぎてノイズに見える |
| v3 | • (丸点) | 0.4em | -0.6em | まだ意図的な装飾に見えにくい |
| v4 | • (丸点) | 0.5em | -0.65em | 認識可能だが、太字感が強い |

### 技術的な知見

- Typst の `regex("[\\p{Han}\\p{Hiragana}\\p{Katakana}ー]")` による CJK 文字検出は正常動作
- `place` + `box` によるオーバーレイは行分割・引用ブロック・リスト内でも破綻しない
- 句読点（。、）は regex に含まれないため自動的にドットが付かない（望ましい挙動）
- 混在テキストでの CJK/Latin 自動分離は正しく機能する

### 結論: 不採用

圏点は和文タイポグラフィとしては正統だが、Markdown の `*emphasis*` の視覚表現としては
不適切と判断した。理由:

1. **意味論の不一致**: 圏点は「太字」に近い強さで読み手に伝わる。italic の「軽い強調」
   という意味論とかみ合わない
2. **スケールのジレンマ**: ドットを小さくするとノイズに見え、大きくすると太字より目立つ。
   「italic 相当の控えめな強調」を圏点で表現できる中間点が存在しない
3. **bold との関係**: `**bold**` より `*italic*` の方が視覚的に目立つのは本末転倒

## 別の発見: emph と h2 の色重複

実験の過程で、emph の色（Pink: #f5c2e7）が h2 見出しと完全に同一であることを発見。
色変更アプローチを維持する場合、emph 用の色を変更すべき。

候補: Lavender (#b4befe / #7287fd) — 実験中に使用し、h2 との区別は良好だった。
