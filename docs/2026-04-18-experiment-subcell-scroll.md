# 実験ノート: サブセルスクロール v1(指数減衰)

2026-04-18。`2026-04-18-design-scroll-animation.md` の設計議論を受けて、その下流補間層の**最小実装**を作り、体感を取ったセッションの記録。後続セッションでパラメータ・アルゴリズムを探索するための前提共有を目的とする。

## 0. 問う問題

- セル単位で離散ジャンプする従来のスクロールを、KGP の `y=src_y` に非セル境界整数値を渡すことで**サブセル(ピクセル)粒度**に変えるのは、視覚的メリットに見合う実装コストか
- コストが妥当として、ごく単純な指数減衰補間で「まず現実性を実証」したときの体感はどうか

## 1. 実装したもの

下流補間層(`target → current` 追従)の最小版。

### 1.1 補間コア

`src/viewer/layout.rs::interpolate_step`

```
current = target + (current - target) * 0.5^(dt / half_life)
```

- フレームレート非依存の閉形式(`2.3.2`)
- `SCROLL_HALF_LIFE_MS = 40`(設計ドキュメントの推奨 80–150ms より攻めた値。理由は §4 参照)
- 残距離 <0.5px でスナップ、`velocity=0` 相当(`3.5`)

### 1.2 状態

`ScrollState`

| フィールド | 型 | 役割 |
|---|---|---|
| `target_y` | `u32` | 目標位置。`ScrollTo` 効果で即座に更新 |
| `current_y` | `f64` | アニメ中の連続位置、毎フレーム `tick` で進む |
| `y_offset` | `u32` | 描画時の離散位置 = `current_y.round() as u32` |

**重要な分離**: 「target は離散イベント駆動で即値」、「current は連続時間ステートマシンで追従」という設計ドキュメント §0 の 2 層モデルをそのまま実装。`y_offset` は表示/計算のための便宜値。

### 1.3 毎 tick の処理

`src/viewer/mod.rs` inner loop 冒頭:

```rust
let dt = now.duration_since(last_tick);
if vp.scroll.tick(dt) {
    vp.dirty = true;
}
```

- `is_animating()` が true の間は `frame_budget`(32ms)で回す
- 停止したら `watch_interval`(200ms)ないし無限長に戻り、再描画ループが消える(§3.5 の省電力条件)

### 1.4 KGP 配置の変更

`src/viewer/viewport.rs` の `ScrollTo` 効果から cell 境界 snap を削除。`y=src_y` に非セル値が入るようになった。

ただし **Split ケース(ビューポートがタイル境界を跨ぐ瞬間)だけは snap フォールバック** を入れている。`top_src_h` が `cell_h` の倍数でないと KGP の `r=行数` と `h=ピクセル` が一致せず、端末が画像を縦方向に圧縮(伸縮アーティファクト)してしまうため。これは設計ドキュメントに記載がない mlux 固有の罠。実装は `display_state.rs::visible_tiles_for_render` に集約。

### 1.5 連打時の累積挙動

`src/viewer/mode_normal.rs` の累積系スクロール(`j`, `k`, `Ctrl-D`, `Ctrl-U`)は、**次の target を `y_offset` ではなく `target_y` から積み上げる** ように変更。設計ドキュメント §3.3 に準じた挙動。

- 変更前: 連打中 → 各キーで「見えている位置から step」を計算するため、前のアニメの遅れぶんだけ累積が過小になる
- 変更後: 連打した回数ぶん target がきっちり積み上がる

回帰テスト: `scroll_accumulates_onto_target_not_render_position`。

絶対 target の `gg`/`G`/`:<n>G` や検索開始位置は意味が違うのでそのまま。

## 2. テスト

- `interpolate_step` の純関数ユニットテスト 5 件(half-life 挙動、フレームレート独立、0dt no-op、収束、逆方向)。設計ドキュメント §3.2(独立性必須)への応答。
- `scroll_accumulates_onto_target_not_render_position`: §3.3 の挙動を固定。
- テスト基盤側 `TestHarness::feed_key` は各キー入力後にアニメを 1 ステップ drain するよう改修。テストは「静止位置」を検証する契約。

## 3. ユーザ観察(2026-04-18 セッション)

- **スクロール中に文字が読みやすくなった**(= 明確な改善の主観)
  - 設計ドキュメント §4.1 の網膜速度 2.5°/s 閾値と整合。離散ジャンプはブラーで読めなかった領域が、サブセル補間で読める領域に入った結果、スクロール中にも情報処理が走るようになった。§4.2 のドリフトテキスト読字モデル(fixation → smooth pursuit)にも合致。
- **半減期 80ms は「遅く」感じた** → 40ms に下げて「軽くなった」
  - 設計ドキュメント §5.2 の推奨初期値(120ms)より攻めた設定。「読む」を軽視せず、かつ即応性を優先した現実解として 40ms が今の着地点。
- **知覚的遅延の存在**: 物理的な到達時間(0 → ~240ms)が延びたのに加えて、「スクロール中も情報処理が走る」ことによる主観時間の延びが乗っている。後者は改善の副作用で、消そうとするとサブセル補間のメリットが失われる。

## 4. 今の到達点と未解決の論点

### 4.1 解けたこと

- サブセルスクロールは視覚的にメリットを生み、実装コストは中程度(変更 ~200 行、テスト追加含めて)で収まった
- 指数減衰 + half_life=40ms で「ひどくない」ベースラインは取れた
- 設計ドキュメント §0 の 2 層分離(target / current)はそのまま実装できた。上流層(`scroll_strategy`, 累積ロジック)は今回ほぼ手付かずで、下流だけで体感が大きく改善した

### 4.2 未解決のまま残っているもの

設計ドキュメントに記載があり、今回は触っていない項目:

- **§3.4 距離適応**: 近距離(`j` 1 発)と大ジャンプ(`gg`/`G`)を同じ half_life で動かしている。近距離は 40ms で十分でも、遠距離は時間不足。`half_life × (1 + log(1 + d/viewport))` 程度の対数スケールを入れる余地。
- **§5.1 初期 ramp-up**: 純粋指数減衰は出だしが一番速い(=初速無限大)。pursuit 立ち上がり潜時 100ms に合わせた ease-in を入れるかどうか。今の体感では不要に見えるが、40ms → より長い half_life に戻す時に問題になるかもしれない。
- **§5.3 減速末尾の延長**: 末尾でさらに half_life を短くして「ピタッと止まる」感覚を作る。現状は 0.5px でスナップしているだけ。
- **§4.6 SDAZ 近似**: 高速スクロール中の描画簡略化。未着手。
- **上流層全般**: EWMA 速度推定、velocity-dependent gain は未着手。今の `scroll_strategy`(`2026-04-12-design-scroll-acceleration.md`)が事実上の上流層。

### 4.3 アルゴリズム選定の再検討余地

指数減衰は「前半速く、後半ダラダラ」のカーブ。読み物に合うかは議論の余地があり:

- **クリティカルダンプばね**: `velocity` 状態を持たせて、末尾で減速するカーブを作れる。オーバーシュートなし(§3.6)。連続 target 更新との相性も良い。
- **cubic-bezier + 固定時間**(Chrome デフォルト型 §2.1): 終了時刻予測可能、末尾がきれいに止まる。ただし連打時に duration リセットが起きて速度不連続(§2.4)。
- **ease-out cubic(末尾だけ)**: 「中間は指数減衰、近接したらカーブ変更」のハイブリッド。

今回の観察「スクロール中に読みやすいからこそ主観時間が延びる」は、「大ジャンプの中間領域で高速に流し、近接で減速する」という設計方針(cubic-bezier 的 or クリティカルダンプばね)を後押しする材料になる。

## 6. 次セッションの選択肢

優先度と独立性で 3 案:

1. **距離適応 half_life**(独立、低リスク、高効用): `j` と `gg` の体感を分離できる。1 行の変更に近い。
2. **アルゴリズム比較**(独立、中リスク、高情報): クリティカルダンプばね or cubic-bezier + 固定時間を別実装し、フィーチャフラグで切り替えて体感比較。読み物用途での「最良」を決め打ちしない。
3. **上流層の EWMA 速度推定**(今回の補間層とは独立、設計ドキュメント §0 の「上流」): 連打速度から target の gain を動的に変える。今の `scroll_strategy` との統合が論点。

個人的な推奨は (1) → (2) → (3) の順。(1) は体感をさらに一段上げつつ、アルゴリズム選定 (2) の判断材料が増える。(3) は上流なのでいつ入れてもよい。
